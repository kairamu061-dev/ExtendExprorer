# BUG-006: 遅いダブルクリックのインラインリネームが不安定・編集ボックスの幅が名前に合わない

- **State**: Fixed
- **Reported**: 2026-07-20
- **Area**: file-list（`FileListView` のインラインリネーム開始判定・編集 TextBox）

## Summary

1. 選択済み項目への「間を空けた 2 回目クリック」でのリネーム開始が安定せず、何かの拍子にしか始まらない。
2. リネーム編集の TextBox の幅が名前にフィットせず、エクスプローラーのように収まらない（固定 `MinWidth=160`）。

## Reproduction

1. ファイルを 1 つ選択し、間を空けてもう一度その名前をクリック
2. 期待どおり編集に入らないことが多い（数回試すと入る）
3. 編集に入っても TextBox が名前より広く/狭く、ぴったりにならない

## Expected

エクスプローラーと同様、選択済み単一項目の再クリックで安定して編集に入り、編集ボックスの幅は名前の長さに合う。

## Cause

1. 開始判定が `PointerPressed` 時点の `List.SelectedItem`（= クリック前の選択）を記録して `Tapped` と突き合わせる方式だった。WinUI の選択確定は概ね pointer 解放時で、この記録タイミングが選択更新と競合し、2 回目クリックの認識が不安定だった。
2. TextBox に固定 `MinWidth=160` を与えており、名前の長さに関係なく一定幅になっていた。

## Fix

1. 開始判定を**タップ完了時点の単独選択項目を候補として覚え、次のタップが同じ項目なら 2 回目クリックとみなす**方式へ変更（`_renameCandidate`）。`PointerPressed` の選択タイミングに依存しなくなり、ダブルクリック（開く）は従来どおり待機満了前の `DoubleTapped` で取り消す（`GetDoubleClickTime()+100ms` 待機）。
2. TextBox の `MinWidth` を撤去し、`TextBlock` で実測した文字幅＋余白（20px、48〜560px にクランプ）を `Loaded`/`TextChanged` で `Width` に設定して名前にフィットさせる。

# BUG-005: 同フォルダへの貼り付けが「- コピー」自動リネームにならず衝突ダイアログになる

- **State**: Fixed
- **Reported**: 2026-07-19
- **Area**: context-menu（`ShellContextMenuService.PasteIntoFolder`）

## Summary

背景メニューの「貼り付け」で、コピー元と同じフォルダへ貼り付けるとエクスプローラーのように
「sample - コピー.txt」が自動生成されず、「送り側と受け側のファイル名が同じです」ダイアログ
（スキップ/キャンセルのみ）が出て複製できない。別フォルダへの貼り付けは正常。

## Reproduction

1. ファイルを右クリック→「コピー」
2. 同じフォルダの空白右クリック→「貼り付け」
3. 衝突ダイアログが表示され、「- コピー」の自動リネームが行われない

## Expected

エクスプローラー同様、同フォルダ貼り付けでは「〜 - コピー」が自動生成される。

## Cause

貼り付けを「フォルダの IDropTarget へのドロップ」で実装しており、ドロップ経路では
衝突時の自動リネーム（`FOF_RENAMEONCOLLISION` 相当）が適用されない。
エクスプローラーの貼り付け（DefView）は、コピー元とコピー先が同じ場合に自動リネームを指定している。

## Fix

貼り付けの実装を IDropTarget ドロップから **`IFileOperation`** ベースへ置換:

1. クリップボードの CF_HDROP からソースパス一覧を取得（`DragQueryFileW`）
2. Preferred DropEffect でコピー/移動を判定（従来どおり）
3. `IFileOperation`（`FOF_ALLOWUNDO`）で Copy/Move。**コピー元の親フォルダ＝貼り付け先のアイテムには
   `FOF_RENAMEONCOLLISION` を付けた別オペレーションで実行**（エクスプローラーと同じ使い分け）
4. 異なるフォルダ間の衝突はシェル標準の衝突ダイアログ（置き換え/スキップ/両方保持）が出るようになる副次改善もある

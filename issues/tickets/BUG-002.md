# BUG-002: ペイン/タブ破棄後にメモリ・ハンドル・スレッドが解放されない（リーク）

- **State**: In Progress
- **Reported**: 2026-07-12
- **Area**: pane-split（`MainViewModel` の `ClosePane`/`CloseTab`、`LayoutHost`/`PaneView`/`FileListView` のライフサイクル・イベント購読解除）

## Summary

分割→分割解消を繰り返すと、1 ペインに戻してもメモリが解放されず単調増加する。WorkingSet だけでなく確定メモリ（Private）・ハンドル・スレッドも増え続け、`GC.Collect()` でも回収されない。ペイン/タブ破棄時に子 View やイベント購読が参照保持されている疑い。

## Reproduction

1. 正式 AOT ビルド（`ExtendExprorer-win-x64`, run `29145714911` / commit `4f4daec`）を Windows 11 で起動
2. 2×2（4 ペイン）へ展開 → 1 ペインへ分割解消、を 1 サイクルとして繰り返す
3. 各サイクル後に **1 ペインに戻した状態**で WorkingSet / Private / Handle / Thread を計測

観測値（すべて 1 ペインに戻した状態）:

| 計測ポイント | WorkingSet |
|---|---|
| リーク試験開始時 | 159 MB |
| 2×2 展開→1 ペインに解消 直後 | 214 MB |
| 4 ペイン展開↔解消 サイクル1後 | 281.6 MB |
| 同 サイクル2後 | 330.6 MB |
| 同 サイクル3後 | 385.4 MB |

- 追加指標（サイクル終盤・1 ペイン時）: **Private 455→459 MB**、**Handle 1156→1168**、**Thread 36→38**
- `GC.Collect()` 実行後も戻らない → WorkingSet の保持ノイズでは説明できない

## Expected

ペイン/タブを閉じて 1 ペインに戻した後、（GC 後に）メモリ・ハンドル・スレッドが概ね試験開始時の水準へ戻る。分割・解消の繰り返しで単調増加しない。

## Cause

`LayoutHost.Rebuild()` がレイアウト構造変更（`SplitPane`/`ClosePane`）のたびに **ツリー全体の `PaneView` を作り直す**が、破棄する古い `PaneView` を購読解除せずに `_paneViews.Clear()` / `RootGrid.Children.Clear()` で捨てていた。

分割・解消では生き残るペイン/タブが多数あり、その**生存 ViewModel（`PaneViewModel` / `TabViewModel`）の `PropertyChanged` / `Tabs.CollectionChanged` イベントが、捨てたはずの古い `PaneView`・`FileListView` を強参照し続ける**。ViewModel はレイアウト木（`MainViewModel.Layout`）に繋がっていて生存するため、古い View は GC されず、分割⇔解消のたびに View（＋ネイティブの `TabView`/`ListView`）が蓄積 → メモリ・ハンドル・スレッドが単調増加していた。

購読経路（`PaneView.ViewModel` セッター / `ObserveActiveTab` / `FileListView.ViewModel` セッター）:
- `PaneViewModel.PropertyChanged` → `OnPanePropertyChanged`
- `PaneViewModel.Tabs.CollectionChanged` → `OnTabsCollectionChanged`
- アクティブ `TabViewModel.PropertyChanged` → `OnTabPropertyChanged`
- `FileListView` 内の `TabViewModel.PropertyChanged` → `OnViewModelPropertyChanged`

## Fix

破棄する `PaneView` を、レイアウト再構築でツリーから外す前に必ず購読解除する。

- `PaneView.Detach()` を追加。`ViewModel = null` を経由して上記 4 経路の購読を解除し、`FileList.ViewModel = null`・`TabItemsSource = null`・`ItemsSource = null` まで連鎖的にクリアする（既存セッターの解除ロジックを再利用）。
- `LayoutHost.Rebuild()` で `_paneViews.Clear()` の前に、既存の全 `PaneView` に対し `Detach()` を呼ぶ。構造変更は全て `LayoutChanged` → `Rebuild()` に集約されるため、分割・ペインクローズ・最終タブクローズのいずれの破棄経路もこの 1 箇所でカバーされる。

これにより生存 ViewModel から古い View への参照が切れ、View と配下のネイティブコントロールが GC 対象になる（ハンドル・スレッドも解放される見込み）。

- **未検証**: ローカル（Linux/WSL）は WinUI ビルド不可のため、CI の AOT 成果物でユーザ環境の再検証が必要（分割⇔解消の繰り返し後に 1 ペインでメモリ・ハンドル・スレッドが戻るか）。

### 第 2 ラウンドの原因分析

Detach で「生存 ViewModel → 破棄 View」の購読経路は切れたが、残存リークは**再構築の作り方そのもの**に起因すると分析:

1. **`LayoutHost.Rebuild()` が構造変更のたびに生存ペイン分も含む全 `PaneView` を新規作成している**。WinUI 3 にはビジュアルツリーから外した要素（特に `TabView`/`ListView`/`ProgressRing` などコンポジションリソースを持つコントロール）が即座に解放されない既知の挙動があり、managed 側の購読を切ってもネイティブ側の蓄積が残る。4 ペイン展開⇔解消の 1 サイクルで PaneView を最大 7 個作り直しており、churn がそのまま増分になる。
2. **`SplitterBar` が生成のたびに `InputSystemCursor.Create()` を呼び、破棄時に Dispose していない**。分割のたびにカーソルリソース（ハンドル）が増える → ハンドル微増と整合。

### 第 2 ラウンドの修正

- `LayoutHost.Rebuild()` を**PaneView 再利用方式**へ変更: レイアウト木に生存しているペインの `PaneView` は使い回し（旧親 Grid から付け替えのみ）、木から消えたペインの View だけ `Detach()` して破棄する。イベント配線は View 生成時の 1 回だけになり、再構築ごとのネイティブコントロール再生成をゼロにする。
- `SplitterBar` のリサイズカーソルを static キャッシュ（縦横 2 個のみ生成）に変更し、生成ごとのカーソルリソース確保をなくす。

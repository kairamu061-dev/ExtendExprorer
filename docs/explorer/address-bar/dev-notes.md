# address-bar 開発メモ

## 実装上の判断

| 判断内容 | 理由 |
|----------|------|
| パンくず＋編集を 1 つの UserControl（AddressBar）に集約し `_editing` フラグで排他 | ペイン内に 1 つ、状態遷移を内部で完結できる。PaneView 側は SetPath と NavigateRequested だけ意識すればよい |
| セグメントはコードビハインドで動的生成（ItemsControl バインドでなく） | 累積パスの計算とクリック配線が単純。セグメント数は可変。`{x:Bind}` テンプレートより見通しがよく AOT も安全 |
| 移動判定を View に持たせず `Func<string,Task<bool>>` で委譲 | PaneView→TabViewModel の既存イベント委譲方式に合わせ、View にファイルシステム知識を持ち込まない |
| 検証は `IFileSystemService.ResolveNavigationTargetAsync`（dir=そのまま/file=親/無効=null） | ファイルパス入力時に親へ移動する spec を 1 メソッドで表現。IO は Task.Run で UI を止めない |
| 余白クリック判定は `e.OriginalSource` からビジュアルツリーを遡って Button を検出 | セグメント（Button）上のクリックは移動、それ以外の余白クリックだけ編集モードにするため |
| Enter 成功時は先に `_editing=false` にしてから表示切替 | 直後の LostFocus による「復帰（revert）」が二重発火して移動を打ち消さないようにする |
